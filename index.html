Character Encodings

Encodings have been used since antiquity to transmit information across distances. However, in most cases the messages were simple or the distances were short, or both. More capable systems were developed and deployed as technology permitted. Notably, the French Chappe optical telegraph, in the early 1790s, used a rooftop apparatus viewed from afar with a telescope and could transmit 98 distinct symbols. But these were used to code entire words by using two symbols, one to choose a code book page and another a word on that page.

Inventors had, from time to time, proposed telegraph systems that encoded individual characters, but the first commercially successful system was the electrical telegraph by the Americans Morse and Vail, which, unlike earlier systems, used only a single wire between stations. In 1844 their system connected Baltimore and Washington, and by 1861 it reached the west coast.

Morse Code

Morse code was developed by Samuel Morse and Alfred Vail and encoded individual letters. The codes were chosen based on the frequency of letters in English text, estimated by the relative counts in the type-cases of a newspaper.

Morse code is efficient for trained human operators, but it was difficult to automate.

ITA 2

The ITA 2 telegraph standard replaced the variable length codes of Morse Code with a fixed-size 5-bit encoding. But since 5 bits aren't enough for all the characters users wanted, two “control codes” were used that shifted between “Letters” and “Figures”, as well as other control codes to perform operations on the receiving teletype like “carriage return” and “line feed”, which may sound familiar.

The code was also organized to spread the hole punches around to reduce and even out wear on electromechanical parts. So the letters were not in the same order as the code points. Note that the letters E and T require one hole and other common letters like AOINSRL take just two holes. The less common letters K, Q, V, and X have four holes punched.

All zeros (no holes punched) is a Null character while all holes punched indicates a shift to Letters, and also Erase or Delete because punching all the holes is a way to correct errors. These uses have continued ever since.

ASCII

ASCII, first introduced in 1963, was well suited to a world with 8-bit bytes, which were used by IBM's System 360 series in 1964 and then rapidly became universal. On 8-hole punched tape, the 8th bit could be used to store a parity bit for error detection. When stored in memory, where no error detection was needed, the high bit was typically set to zero.

Microcomputer makers of the 1970s made various ASCII extensions that used the high 128 values of a byte, and sometimes the nonprintable ASCII code points for various purposes like line and box drawing characters for simple graphical user interfaces, accented letters, computer games, etc.

In 1987 ISO attempted to standardize some of this. That resulted in ISO/IEC 8859-1 (Usually shortened to ISO-8859-1) and other similar character sets, including Windows-1252. These added support for various western languages' accented characters and punctuation.

But 8 bits is still not enough to encode all the characters users wanted. Which leads us to Unicode.

Unicode

Development of Unicode started in the late 1980s and initially envisioned a 16-bit code, which would be sufficient to encode characters published in modern text.

In the original 16-bit Unicode, the first 256 code points were taken from ISO-8859-1.

A 16-bit code still wasn't enough, though, so with Unicode 2.0 in 1996, support was added for over a million code points. Including, these days, various emoji. A “code point” is often a character, but the standard allows for the use of separate code points with combining characters, such as diacritical marks to add to letters or skin color modifiers to change the appearance of human emojis.

Obviously most of these codes are not used in any given document, so it would be hugely wasteful to store the code points themselves.

Instead, various Unicode Transformation Formats were developed, including UTF-8, UTF-16, and UTF-32.

UTF-32 encodes each code point in 32 bits (4 bytes), with at least 11 leading zero bits. This greatly simplifies decoding at the expense of using 4 bytes for each character, no matter how common. UTF-16 encodes most commonly-used code points as a single 16-bit value. All other code points are encoded in two 16-bit values. UTF-16 is unpopular. Which brings us to UTF-8.

UTF-8

UTF-8, however, is by far the most popular transformation format. It has been implemented in all major operating systems available today. It has various advantages.

  • It is 100% backward compatible with 7-bit ASCII (the first 128 Unicode code points).
  • It is easy to synchronize by simply examining the high-order bits in a byte stream
  • Encoding and decoding are relatively straightforward.
Encoding results in sequences of 1 to 4 bytes.
  • The first 128 code points (corresponding to 7-bit ASCII) use 1 byte.
  • A further 1,920 code points use 2 bytes.
  • The remaining 61,440 code points on the “Basic Multilingual Plane” use 3 bytes.
  • All other code points require 4 bytes.
It does have some minor disadvantages.
  • As a variable length code it may be necessary to back up or skip forward between 1 and 3 bytes to find the first byte of an encoded code point.
  • It loses efficiency if code points outside the Basic Multilingual Plane are heavily used. Emojis, as an example, are in this category of code points.
  • Not all byte sequences are valid UTF-8. Decoders must handle error conditions due to erroneous sequences such as unexpected continuation bytes, lack of an expected continuation byte, sequences that don't decode to valid Unicode code points, and so on.

Bits are stored highest to lowest, with leading zeros if needed. The first UTF-8 byte has the highest bits in the code point and the last UTF-8 byte contains the lowest bits.

When encoding Unicode code points to UTF-8, the encoder just checks which of the four UTF-8 encodings to use based on the code point value.

When decoding UTF-8 to Unicode code points, the decoder knows what to do by simply looking at the most significant bits in a UTF-8 byte.
